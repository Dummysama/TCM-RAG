# 基于大模型的中医药智能问答系统（TCM-ARG）
> Large Model-Based Intelligent Question-Answering System for Traditional Chinese Medicine

TCM-ARG 是一个面向中医药场景的智能问答系统，基于 **大语言模型（LLM）** 与 **检索增强生成（RAG, Retrieval-Augmented Generation）** 架构构建。系统围绕中药、方剂（含中成药）、证候、症状等中医药知识开展问答服务，通过“**知识检索 + 证据约束生成**”的方式，提高回答的准确性、可解释性与可用性。

与直接调用通用大模型不同，TCM-ARG 会先在本地中医药知识库中检索相关证据，再将检索结果与用户问题一同输入模型生成答案，从而有效降低幻觉风险，并支持引用展示与知识卡片交互。

---

## 1. 项目背景

中医药知识体系具有术语专业、表达凝练、知识分散、用户理解门槛较高等特点。传统关键词检索方式虽然能够返回相关条目，但难以满足用户以自然语言直接提问、获取结构化解释的需求。

近年来，大语言模型在自然语言理解与生成方面取得了显著进展，为智能问答系统提供了新的技术路径。但在中医药等垂直领域，直接使用通用模型仍存在以下问题：

- 缺乏专业知识支撑，容易出现事实性错误或“幻觉”
- 回答结果缺少可追溯证据，可信度不足
- 对中医药术语、方剂名称和症状描述的理解存在偏差
- 难以兼顾问答自然性与专业性

为解决上述问题，项目采用 **RAG 架构**：在答案生成之前，先对问题进行意图识别与实体分析，再从本地知识库中召回相关证据，最后基于证据生成回答，实现“**先检索、后生成**”的证据约束问答流程。

---

## 2. 项目目标

本项目旨在构建一个面向中医药领域的智能问答系统，使用户能够通过自然语言方式查询中医药相关知识，并获得相对准确、清晰且具有证据支撑的回答。

系统目标包括：

- 支持中药、方剂、症状、证候等多类型问答
- 构建结构化中医药知识库并完成向量化索引
- 实现基于语义检索的 RAG 问答流程
- 提供中药与方剂知识卡片，增强问答结果的可解释性
- 支持用户注册、登录、历史会话管理与持久化存储
- 提供 Web 前端交互界面，支持本地运行与服务器部署
- 对高风险问题进行安全拦截与就医提示

## 3. 数据来源

系统知识库主要基于中医药结构化表格数据构建，覆盖以下内容：

- **中药基础信息**：药材中文名、拼音、拉丁名、英文名、性味、归经、功效、主治、别名等
- **方剂与中成药信息**：方剂名称、组成中药、主治症状、主治证候、来源等
- **中药—化合物关联数据**
- **化合物—靶点关联数据**
- **中医证候数据**
- **中医症状数据**
- **中药转录组学数据**
- **实体关系说明文档**

这些数据经过清洗、规范化、字段映射、文本拼接与知识切片后，构成系统的本地知识库与向量检索基础。

## 4. 系统功能概述

当前系统已实现以下核心功能：

### 4.1 中医药问答
- 支持中药属性问答，如中药功效、性味、归经、主治等
- 支持方剂问答，如组成、功效、适用证候等
- 支持基于症状描述的推荐类问答
- 支持证据约束回答，降低生成式问答的幻觉风险

### 4.2 问题理解与路由
- 对用户问题进行预处理
- 识别问题意图
- 提取实体信息
- 区分中药类、方剂类、推荐类等问题类型

### 4.3 知识卡片
- 回答结果支持展示引用标签
- 支持点击中药标签弹出中药详情卡片
- 支持点击方剂标签弹出方剂详情卡片
- 支持展示名称、拼音、组成、功效、主治等信息

### 4.4 用户系统与会话管理
- 用户注册与登录
- JWT 登录态校验
- 历史会话持久化
- 不同用户数据隔离
- 刷新页面后恢复历史问答记录

### 4.5 风险问题拦截
- 对胸痛、呼吸困难、孕期高风险、儿童抽搐等高风险场景进行安全提示
- 对非中医药或无法识别的问题进行友好输入引导

## 5. 系统整体架构

系统整体采用前后端分离设计，主要由以下模块组成：

### 5.1 数据处理模块
负责原始中医药数据的清洗、规范化、字段映射、文本拼接与知识切片，形成适合检索与生成的知识单元。

### 5.2 知识库与索引模块
负责构建中医药知识库，并通过嵌入模型将知识片段向量化，建立本地向量索引。

### 5.3 检索模块
接收用户问题后完成语义检索，从知识库中召回相关中药、方剂或其他知识片段，并构建证据块。

### 5.4 大模型生成模块
基于检索到的证据块调用大语言模型生成回答，输出结构化问答结果与引用信息。

### 5.5 后端服务模块
基于 FastAPI 提供统一 API，包括问答接口、用户接口、会话接口以及知识卡片详情接口。

### 5.6 前端交互模块
基于 React 实现问答页面、会话列表、登录界面、知识卡片弹窗、移动端适配等交互功能。

### 5.7 部署与运行模块
负责本地开发运行、服务器部署、服务管理以及前后端联调。

## 8. 核心工作流程

系统问答流程如下：

1. 用户在前端输入自然语言问题
2. 前端调用后端问答接口
3. 后端进行问题预处理、意图识别与实体抽取
4. 后端根据问题类型执行本地知识检索
5. 检索结果被组织为证据块（evidence blocks）
6. 将用户问题与证据块共同输入大语言模型
7. 大语言模型在证据约束下生成回答
8. 返回答案、引用信息与知识卡片入口
9. 前端展示结果，并支持用户点击标签查看详情

## 10. 本地开发运行

### 10.1 克隆项目
```bash
git clone https://github.com/yourname/TCM-RAG.git
cd TCM-RAG
```

### 10.2 创建并激活虚拟环境
```bash
python3 -m venv .venv
source .venv/bin/activate
```

### 10.3 安装后端依赖
```bash
pip install -r requirements.txt
```

若未维护完整的 `requirements.txt`，可按实际依赖手动安装：

```bash
pip install fastapi uvicorn sqlalchemy "python-jose[cryptography]" "passlib[bcrypt]" python-multipart faiss-cpu sentence-transformers openai
```

### 10.4 启动后端
```bash
uvicorn api_server:app --host 127.0.0.1 --port 8000 --reload
```

### 10.5 安装前端依赖
```bash
cd tcm-rag-frontend
npm install
```

### 10.6 启动前端
```bash
npm run dev
```

### 10.7 浏览器访问
```text
http://localhost:5173
```

## 11. 服务器部署

### 11.1 上传项目
可通过 Git 拉取，或使用 `scp` 上传前后端文件。

### 11.2 安装依赖
```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

### 11.3 启动后端服务
开发调试可直接运行：

```bash
uvicorn api_server:app --host 0.0.0.0 --port 8000
```

正式环境建议使用 `systemd` 管理服务。

### 11.4 Nginx 反向代理
将 80 端口请求代理到本地 `127.0.0.1:8000`，用于统一访问入口。

### 11.5 前端构建
```bash
cd tcm-rag-frontend
npm run build
```

构建后的 `dist` 可由后端静态挂载，或由 Nginx 独立托管。
